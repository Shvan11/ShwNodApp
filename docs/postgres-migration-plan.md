## Context

**⚠️ Branch — remote/cloud agent READ FIRST:** Ground this plan on branch **`dolphin-native`** of `https://github.com/Shvan11/ShwNodApp`, **NOT `main`**. `main` is behind and **stale** — it still contains `services/database/queries/dolphin-queries.ts` + `routes/api/dolphin.routes.ts` and the old production `init_script.sql`, so if you see those, you're on the wrong branch. On `dolphin-native` the DolphinPlatform decoupling is **complete** (those files are deleted) and `init_script.sql` is the refreshed live dump. **Every baseline claim in this plan is verified true on `dolphin-native`.** If your session cloned the default branch, run `git checkout dolphin-native` and re-ground before refining. Local working copy: `C:\ShwNodApp-dolphin`.

**Why:** The app runs on SQL Server (mssql v12 / tedious) with raw T-SQL spread across 21 query modules plus 39 stored procedures, 25 triggers, and 4 scalar functions living *inside* the database. The team wants to move to **PostgreSQL**, partly to enable the planned move to a **Linux-based server** (SQL Server is a Windows-centric dependency) and partly to consolidate the stack (a Postgres footprint already exists via the Supabase aligner-sync bridge).

**Why now:** The `dolphin-native` branch just **completed the decoupling from the third-party DolphinPlatform SQL Server database** — all six cross-DB stored procs were dropped, `dolphin-queries.ts`/`dolphin.routes.ts` deleted, and the data cloned into local tables. That removed the one hard blocker: the app no longer reads any database it doesn't own, so **no parallel SQL Server connection is needed after migration**. This is the clean moment.

**Intended outcome:** A PostgreSQL-backed app where (1) the data layer is a **typed Kysely query builder** (no raw dialect drift, compile-time types), (2) **all business logic lives in TypeScript** (no procs/triggers/functions in the DB — portable and testable), (3) the DB runs as a **local native service** (LAN-fast, works offline, like SQL Server does today), proven on the `ShwanNew_Test` sandbox **before** production (`ShwanNew`) is touched.

**Locked decisions (chosen by the user):**
| Decision | Choice |
|---|---|
| Hosting | **Local native PostgreSQL** on the clinic server (offline-capable; not cloud/Supabase) |
| Data layer | **Kysely** typed query builder replacing the raw-SQL facade + 21 modules |
| Procs/triggers/functions | **Move ALL logic into TypeScript services**; retire all DB-side logic |

## Authoritative current state (from live `ShwanNew_Test` introspection; `init_script.sql` was refreshed from the live DB on 2026-05-28 — see Phase 0, ✅ done)

- **68 tables**, 20 views, **39 stored procedures**, 4 scalar functions, **25 triggers**, 2 TVP table-types, 62 IDENTITY cols, 7 computed cols, 58 FKs, 26 checks, 82 defaults, 274 non-PK indexes.
- **Data volume ~266k rows total** (largest: `tblInvoice` 82k, `tblappointments` 67k, `tblvisits` 34k) — ETL is **minutes**, never the bottleneck.
- **3 findings that drive the risk profile:**
  1. **25 triggers** (snapshot showed 0): ~8 Supabase-sync enqueue (`trg_sync_*` → `SyncQueue`), ~6 aligner sequencing-logic on `tblAlignerSets`, the rest patient-type/photo/employee bookkeeping. Hidden logic that must move to TS.
  2. **All 201 string columns are `Arabic_CI_AS`** (case-INsensitive, accent-sensitive). PG is case-SENSITIVE by default → every text `WHERE`/`JOIN`/`LIKE`/unique silently changes unless handled.
  3. **`tblappointments.SSMA_TimeStamp` is a `rowversion`** (MS-Access-via-SSMA artifact) with no PG equivalent — **verified no app consumer → drop it.**
- Data-access today: facade `services/database/index.ts` (`executeQuery`/`executeStoredProcedure`/`withTransaction`/`withRequest` + re-exported `sql`/`TYPES`), `@name` params as `[name, MssqlType, value]` tuples, **positional `columns[N].value` row mappers**, `useUTC:false`, single tarn pool (`pool.ts`). ~30 route/service files consume the query functions. No ORM, no migration runner.

## Highest-risk items & de-risking
1. **Collation** (`Arabic_CI_AS`→PG case-sensitivity) → `citext` (lowest churn, preserves accent-sensitivity) + ICU for ordering; dedicated collation parity suite run early.
2. **Trigger reimplementation** (esp. aligner derived-state) → reimplement as the transactional write path, port aligner first, diff resulting table *state*, centralize all writes through services.
3. **Positional-mapper rewrite (~8,300 lines)** → the Phase 3 bridge keeps mappers working during the driver swap; delete them incrementally in Phase 4, never big-bang.
4. **Date/`useUTC:false`** → `timestamp` not `timestamptz`, `date`→string parser centralized (done in Phase 1's `kysely.ts`), ETL reuses the app's own coercion.
5. **Sync-trigger removal (prod)** → transactional app-level enqueue (Option A); sandbox neutralized so it never blocks the proof.
6. **BIT→boolean `=== 1` assumptions** → Kysely `boolean` types catch most at compile time; audit bit-bearing paths.
