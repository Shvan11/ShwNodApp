## 1. Executive summary

The seam is **functionally working but structurally fragile**. Verified-clean: endpoint
reachability (every sampled method/path pair matches, no hardcoded URLs), SSE event naming, and
the PG type-parser strategy (`date`→`'YYYY-MM-DD'` string, `numeric`/`bigint`→number,
`bit`→boolean). The core problem is **there is no enforced contract at the boundary**:

- The backend emits at least **three** response shapes — raw data, the `sendSuccess` envelope,
  and per-endpoint custom objects.
- The hand-maintained FE/BE types have **drifted** (phantom root-level `code`, missing `timestamp`).
- **310 raw `fetch()` calls across 85 files bypass `public/js/core/http.ts`**, so error handling,
  credential transmission, and response validation are all per-component.
- Only the Patient Portal validates responses (Zod at the fetch boundary); the staff app trusts
  the wire.

Net effect: a broken query or renamed field surfaces as `undefined` in the UI instead of a caught
error.

**Only two findings are currently user-visible bugs:** H7 (dates render a day early) and H8
(chair-display visit date never renders). M1 is degraded-but-not-silent UX. Everything else is
structural risk that hasn't bitten yet.

## 2. Recommended remediation sequence

Work top-down — later phases assume earlier ones landed.

1. **Quick wins** (§4) — the user-visible bugs (H7, H8, M1) + cheap type/mount fixes. Do first.
2. **Standardize the backend envelope** — every route on `sendSuccess`/`ErrorResponses`; unify
   `code` location; always emit `timestamp`. Items: **H4, H9, M1, M2, M5**.
3. **Fix FE types** to match the now-stable contract. Items: **M2, M3**.
4. **Introduce `@tanstack/react-query` + an ESLint ban on bare `fetch`**, migrating high-traffic
   components first. Items: **H1, H3, M4, M7, M8**.
5. **Add Zod validation** — response validation at the staff fetch boundary (**H11**) and
   `validate()` on mutations (**H10**).
6. **CSRF** protection on mutations (**H2**) — best layered in once everything routes through one
   client (step 4).
7. **Optional / last:** `ts-rest` or zod-openapi codegen to make the contract self-enforcing
   (only after the envelope is stable).

> **⚠️ Sequencing correction (2026-06-04, session 3 — see N9). Steps 2 and 4 are in the wrong
> order for safety; do H1 before H4.** `core/http.ts` does **not** unwrap the `{success, data}`
> envelope — it returns the raw body — so flipping a GET route to `sendSuccess` (H4) changes what
> *every* caller receives (`data` → `{success, data, …}`), breaking the 310 raw fetches that read
> the body directly. There is no safety net to do H4 "first." **Corrected order:**
> **(a)** make `core/http.ts` envelope-tolerant — return `body.data` for a `{success:true, data}`
> envelope, else pass the body through (a **no-op today**, since no route is enveloped yet);
> **(b)** funnel callers through it (H1) — behavior-preserving, so each migration is verifiable in
> isolation; **(c)** *then* flip routes to `sendSuccess` (H4), now transparent to funneled callers.
> All three **incremental, endpoint-by-endpoint — never big-bang.** The localized Phase-2 envelope
> items that *don't* change a success-path body shape (H9, M1, M5-partial) were safe to land ahead
> of this and are done. **Decisions for the user before the H1/H4 migration:** adopting
> `@tanstack/react-query` (new dependency) and an ESLint `no-restricted-syntax` ban on bare
> `fetch()` are policy/architecture calls — surface them, don't land them unilaterally.
>
> **Update (session 4): step (a) is DONE — see H1 Done note + N10.** It was **not** the no-op N9
> predicted: the file-explorer routes are already enveloped and already funneled through
> `core/http.ts`, so the 4 affected callers were migrated atomically with the unwrap. **(b)** (funnel
> the remaining raw fetches) and **(c)** (flip routes to `sendSuccess`) are still pending, gated on
> the two user decisions above.
>
> **Update (session 12): step (b) is DONE — H1 → `[x]`.** All ~310 raw `fetch()` sites are funneled
> through `core/http.ts` (or are documented inline-disabled stay-raw exceptions), the ESLint bare-`fetch()`
> ban is promoted to **`error`**, and H3 (credentials on mutations) fell out resolved. **Only (c) / H4
> remains** — flipping GET routes to `sendSuccess`, which is now **transparent** to every funneled caller
> *and* to the route loaders (`apiLoader` unwraps as of session 12). The React-Query layer (the *staged*
> half of the session-4 decision — RQ on top of the funnel, starting with the appointments screen) is a
> separate enhancement that now sits under **M7/M8** (timeout/retry/abort/cache), not a blocker for H4.

### N14 — Tolerant `Promise.all` is a 4th funnel mechanic; +1 enveloped family; an `Array.isArray`/`unknown[]` typing gotcha (2026-06-04, session 7)

### N15 — useStand done; the clean-mechanical tail is exhausted; a consumer-side message-preservation mechanic (2026-06-04, session 8)

- **Why H1-before-H4 paid off (N9 vindicated).** Because every FE caller now funnels through `core/http.ts` (which `unwrapEnvelope`s `{success,data}`) **and** the route loaders go through `fetchData` (session 12), flipping `res.json(data)` → `sendSuccess(res, data)` is **invisible** to the whole app — *if* the consumer is funneled and the value is non-null.   - **(a) bare data** `res.json(x)` → `sendSuccess(res, x)`. The consumer already reads `x` directly; unwrap returns `x`. ✅ when `x` is non-null.
  - **(b) `{success, …keys}`** (no `data` key) → `sendSuccess(res, {…keys})`. Post-unwrap the FE reads the same `.keys`. ✅ **iff** no consumer reads top-level `.success` (grep each).
  - **(c) `{success,[message],data}`** (has `data` key) → `sendSuccess(res, data, message)`. **Always** ✅ — the *current* unwrap already returns just `data` (success+message dropped today), so the post-unwrap value is byte-identical; only an unread `timestamp` is added.
  - Plus **ignored-return mutations** (any shape) and **hand-rolled non-2xx errors → `ErrorResponses.*`** (FE reads `err.data.error`).
- **The `sendSuccess(res, null|undefined)` data-omit hazard.** `error-response.ts:87` **omits the `data` key** when the payload is null/undefined, so the body is `{success,timestamp}` (no `data`) → `unwrapEnvelope` finds no `data` key → **passes the envelope through** → a funneled caller expecting `null` now gets `{success,timestamp}`. So a bare-data GET whose value can be `null` (e.g. patient `/info` if not-found, a `getX()` single-object query) is **not** a safe (a)-conversion without a null guard. **Arrays are always safe** (Kysely returns `[]`, never null); guaranteed-non-null objects (`{x: a||''}`, `row ?? {…}`) are safe; nullable single-object reads are **deferred** until a null-guard/404 is confirmed.
- **"The funnel is done" ≠ "every consumer is funneled."** Two route families are read by **non-funneled** clients that parse the **raw** body, so the transparent rule does **not** apply: `whatsapp.routes.ts` + `messaging.routes.ts` go through the bespoke `whatsapp-api-client.ts` (its own validate/retry/abort), and the **portal** is the Zod boundary (N17). For those, only **wire-identical** conversions are safe (e.g. messaging `/count`+`/reset` already emit the full `{success,data,timestamp}` shape, so they're conformant and untouched). **Always re-grep the concrete consumer set per route** before trusting transparency — don't generalize from "H1 closed."
